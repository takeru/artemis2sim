patched-conic で free-return を作るなら、月到着直前の Moon-relative 超過速度 \(v_\infty\) と perilune 半径 \(r_p = R_M + h_p\) から、月周り双曲線の離心率と turning angle を

```text
e = 1 + (r_p v∞²)/μM
δ = 2 asin(1/e)
```

で出し、到着 \(v_{\infty^-}\) を flyby plane で角度 \(δ\) だけ回して出発 \(v_{\infty^+}\) を作り、これを Earth-centered に戻して地球帰還弧を伝播します。最終的には **EI 時刻・EI 高度・return leg の到着幾何** が official baseline と合うように、`TLI state` と `flyby plane` を数値的に解くのが実務的です。Artemis II の公開資料は free-return、flyby 高度、EI/Splashdown MET を与えているので、この逆算は十分可能です。 citeturn412443view2turn147277view0turn221860view0

**イベント検出式**も先に決めておくと実装が楽です。月最接近は `d(t)=|r-rM|` の局所最小、月重力圏入り/離脱は patched-conic なら SOI 半径との交差、n-body なら press kit の official MET を比較基準にして event detector を合わせます。通信断は、Earth と spacecraft を結ぶ線分が Moon 球と交差するかで判定できます。つまり、線分 `E→SC` に対する Moon 中心からの最短距離が \(R_M\) 未満で、最近接点パラメータが区間 \([0,1]\) にある時間帯を blackout とします。これで press kit の **30〜50分**と比較できます。EI は press kit のとおり **400,000 ft** 高度で切ればよいです。 citeturn619818search0turn619818search2turn221860view0

**正解データの取り方**としては、JPL Horizons をそのまま使うのが最短です。Horizons API は **GET で `EPHEM_TYPE=VECTORS` を指定すると Cartesian state vectors を返せる**ので、まずそれで Artemis II の時系列ベクトルを取得し、そこから closest approach、blackout、EI、splashdown 近傍のイベントを再計算するのがよいです。Horizons Lookup API は `sstr` に **spacecraft** を渡せて、`group=sct` で宇宙機だけに絞れます。さらに JPL の検索結果スニペットでは、**Artemis II spacecraft は “Artemis II / Integrity / Orion EM-2”、ID は -1176** と読めます。つまり、**まず Horizons で -1176 を truth trajectory として読む**、その上で what-if 用の自前 free-return solver を別に持つ、という二層構成がいちばん堅いです。 citeturn572743view0turn857322view1turn857322view0turn572743view2turn857322view3turn366588search0turn366588search1

API 叩きの形はこんな感じです。

```bash
curl "https://ssd.jpl.nasa.gov/api/horizons.api?format=text\
&COMMAND='-1176'\
&OBJ_DATA='YES'\
&MAKE_EPHEM='YES'\
&EPHEM_TYPE='VECTORS'\
&CENTER='geo'\
&START_TIME='2026-04-01 22:24'\
&STOP_TIME='2026-04-11 23:00'\
&STEP_SIZE='1 min'\
&VEC_CORR='NONE'"
```

`CENTER` は geocenter/body-center を取れますし、`VECTORS` は dynamical studies/programming 向け、`VEC_CORR=NONE` は幾何学的 state を返す設定です。Lookup で名前解決してから本問い合わせを組む流れで問題ありません。 citeturn857322view0turn857322view1turn572743view2

実装方針としては、**段階を分ける**のが安全です。  
まず **Stage 1** で official MET をそのまま使うイベント予測器を作る。

この順なら、途中でどこがズレているかを常に official data と突き合わせられます。 citeturn221860view0turn366588search0turn572743view0
