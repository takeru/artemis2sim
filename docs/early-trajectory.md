# T+0 〜 T+3.4h 合成軌道ドキュメント

## やりたいこと

JPL Horizonsのエフェメリスデータはミッション開始約3.4時間後（T+3.4h）からしか存在しない。
T+0（打ち上げ）からT+3.4hまでの軌道を物理シミュレーションで合成し、
Horizonsデータと滑らかに接続する。

## ミッション概要（T+0 〜 T+3.4h）

この区間で宇宙船は以下を行う:
1. 大気圏内を上昇（打ち上げ）
2. 投入軌道に入る
3. 2回のICPSバーンで高楕円チェックアウト軌道に遷移
4. ICPS分離
5. 近接運用デモ

## 現在の実装

### 処理の流れ（gen_trajectory.py の synthesize_early_trajectory）

```
1. KSC位置計算（ECEF→ICRF、ERA回転）
2. 軌道面・基底ベクトルの決定（Horizonsデータから）
3. 投入軌道パラメータの設定
4. 9パラメータ最適化（scipy Nelder-Mead）
5. 最適化されたパラメータでポイント生成
   Phase 0: 打ち上げ上昇（T+0 → ASCENT_H）
   Phase 1: 投入軌道（ASCENT_H → BURN1_H）
   Phase 2: バーン1後 → バーン2（BURN1_H → BURN2_H）
   Phase 3: バーン2後 → Horizonsデータ開始（BURN2_H → first_met_h）
6. 最終点をHorizonsの最初の点に接続
```

### 9つの最適化パラメータ

| # | パラメータ | 意味 | 初期値 | 境界 |
|---|-----------|------|--------|------|
| 1 | dv1 | バーン1 プログレードΔv (km/s) | 0.017 (vis-viva計算値) | -0.5 〜 1.0 |
| 2 | dv2 | バーン2 プログレードΔv (km/s) | 2.276 (vis-viva計算値) | 1.5 〜 3.5 |
| 3 | dv1r | バーン1 ラジアルΔv (km/s) | 0 | -0.3 〜 0.3 |
| 4 | dv2r | バーン2 ラジアルΔv (km/s) | 0 | -0.3 〜 0.3 |
| 5 | azimuth | 打ち上げ方位角 (rad) | 0 | -0.2 〜 0.2 |
| 6 | ins_peri | 投入軌道 近地点 (km) | R_E + 27 | R_E-200 〜 R_E+200 |
| 7 | ins_apo | 投入軌道 遠地点 (km) | R_E + 2222 | R_E+500 〜 R_E+5000 |
| 8 | b1h | バーン1 有効時刻 (hours) | 0.82 | 0.6 〜 1.1 |
| 9 | b2h | バーン2 有効時刻 (hours) | 1.8 | 1.5 〜 2.2 |

### 制約（合わせるべき値）= 6

Horizonsデータの最初の点の状態ベクトル:
- x, y, z（位置3成分）
- vx, vy, vz（速度3成分）

### コスト関数

```python
cost = position_error_km + velocity_error_km_s * 1000
```

位置誤差(km) + 速度誤差(km/s)×1000。速度1 m/sの誤差 = 位置1kmの誤差と等価。

### 物理モデル

- 地球2体問題（点質量重力）
- 月の重力（3体問題、潮汐力項含む）
- RK4積分、10秒ステップ
- インパルスバーン（瞬間的な速度変化）

含まれていないもの:
- J2（地球扁平率）: 調査済み、3.4hで~400kmの効果。ただし実装すると最適化結果が悪化（41km→51km）。原因未特定（ICRF Z軸と地球自転軸の微小なずれ、またはモデル間の相互作用）。コメントアウトして無効化中。
- 太陽重力: 3.4hで~0.04km。無視できる。
- 大気抵抗: 近地点通過時のみ。~0 m/sの速度ロス。無視できる。
- 有限バーン時間: 遠地点上昇は18分間燃焼だがインパルス近似。バーン時刻の最適化で「有効中心時刻」として吸収。

|---|------|--------|------|------|
| 1 | 接続ジャンプ | ✅解決 | 0km | propagate()と出力コードの構造一致で解決 |

### 解決済みの問題の経緯

#### 接続ジャンプ（#1）の解決過程
1. 最初: 逆伝播（backward RK4）→ 地球の反対側にジャンプ（10,000km+）
2. Lambert解法 → 2体問題の限界で7,200kmジャンプ
3. 2バーン射撃法（グリッド探索）→ 913km
4. scipy最適化（5パラメータ）→ 47km（速度方向3.7°ずれ）
5. 9パラメータ最適化 → 0km/0°だが出力コードで224kmジャンプ
6. propagate()と出力コードの2区間構造を一致 → **0km解決**

#### S字カーブ（#5）の解決過程
1. Phase0（線形上昇）→ Phase1（ケプラー楕円、近地点-200km）で高度が急降下
2. 2分間ブレンド導入 → 軽減だが完全ではない
3. MECO高度160km固定 + ブレンド改善 → ほぼ解消

### ファイル構成

- `gen_trajectory.py`: 軌道合成スクリプト（synthesize_early_trajectory関数）
  - scipy.optimize.minimize (Nelder-Mead) による9パラメータ最適化
  - RK4積分（地球+月重力、10秒ステップ）
  - ケプラー方程式ソルバー（投入軌道Phase 1用）
  - .venv内のscipy必要（`uv venv && uv pip install scipy`）
- `trajectory.json`: 生成された軌道データ（合成+Horizons結合）
- `data/horizons/`: JPL Horizonsの生データ
  - `fetch.sh`: 再取得スクリプト
  - `spacecraft.txt`, `moon.txt`: 10分間隔データ（全ミッション）
  - `spacecraft_fine.txt`, `moon_fine.txt`: 1分間隔データ（最初の28h）
- `debug_burns.py`: バーンデバッグ用スクリプト

### 改善案（未実装）

優先度高:
- **打ち上げ上昇の物理シミュレーション**: 推力+大気抵抗+重力ターンモデル。これが解決すれば#2,#3,#4,#8が同時に解消する可能性。ただし実装が複雑（大気密度モデル、SLS推力プロファイル、空力係数が必要）
- **Horizonsデータの定期再取得**: ミッション中にエフェメリスが更新される。実飛行データが全区間カバーすれば合成自体が不要になる

優先度中:
- **gen_trajectory.py高速化**: Numba @njitでRK4ループを高速化（10-50倍）。または初期値キャッシュで再実行時にscipy評価回数を削減
- **有限バーン時間のモデル化**: 18分間の遠地点上昇を分割して伝播。バーン時刻の最適化が不要になる可能性
- **J2の調査・修正**: ICRF座標系での正確なJ2実装の検証

優先度低:
- **近地点引数（M_offset）の最適化**: 10番目のパラメータとして追加可能だが、現状0°で収束しており効果は限定的
- **太陽重力の追加**: 3.4hの伝播で0.04kmの効果。無視できる

## 公開値との検証結果（2026-04-02時点）

### 合成区間（Phase 0〜3, T+0 → T+3.4h）

最適化でHorizonsデータのT+3.4h地点に0km/0°で接続するが、
途中の軌道パラメータは公開値と乖離している。
これは打ち上げ上昇をケプラー楕円で近似していることの限界。

| 項目 | シミュレータ | 公開値 | ソース | 判定 |
|------|------------|--------|--------|------|
| 投入軌道 近地点 | 72km | 27km | SpaceNews | ❌ +45km |
| 投入軌道 遠地点 | 3,526km | 2,222km | SpaceNews | ❌ +1,304km |
| burn1後 近地点 | -8km | 185km | CBS, SpaceNews | ❌ 非物理的 |
| burn1後 遠地点 | 3,397km | 2,222km | 推定 | ❌ +1,175km |
| burn2時の高度 | -5km | ~185km | 推定 | ❌ 非物理的 |
| T+3.4h接続 | 0km / 0° | — | — | ✅ |

原因: 最適化パラメータ（特に投入軌道近地点-200km）が非物理的な値に収束し、
途中経路が実際とずれる。ただし終点（Horizonsデータとの接続）は正確。
