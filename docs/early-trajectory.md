# T+0 〜 T+3.4h 合成軌道ドキュメント

## やりたいこと

JPL Horizonsのエフェメリスデータはミッション開始約3.4時間後（T+3.4h）からしか存在しない。
T+0（打ち上げ）からT+3.4hまでの軌道を物理シミュレーションで合成し、
Horizonsデータと滑らかに接続する。

## ミッション概要（T+0 〜 T+3.4h）

この区間で宇宙船は以下を行う:
1. 大気圏内を上昇（打ち上げ）
2. 投入軌道に入る
3. 2回のICPSバーンで高楕円チェックアウト軌道に遷移
4. ICPS分離
5. 近接運用デモ

## 現在の実装

### 処理の流れ（gen_trajectory.py の synthesize_early_trajectory）

```
1. KSC位置計算（ECEF→ICRF、ERA回転）
2. 軌道面・基底ベクトルの決定（Horizonsデータから）
3. 投入軌道パラメータの設定
4. 9パラメータ最適化（scipy Nelder-Mead）
5. 最適化されたパラメータでポイント生成
   Phase 0: 打ち上げ上昇（T+0 → ASCENT_H）
   Phase 1: 投入軌道（ASCENT_H → BURN1_H）
   Phase 2: バーン1後 → バーン2（BURN1_H → BURN2_H）
   Phase 3: バーン2後 → Horizonsデータ開始（BURN2_H → first_met_h）
6. 最終点をHorizonsの最初の点に接続
```

### 最適化パラメータ（9個、うち2個はソフト制約）

| # | パラメータ | 意味 | 初期値 | 境界 |
|---|-----------|------|--------|------|
| 1 | dv1 | バーン1 プログレードΔv (km/s) | 0.017 (vis-viva計算値) | -0.5 〜 1.0 |
| 2 | dv2 | バーン2 プログレードΔv (km/s) | 2.276 (vis-viva計算値) | 1.5 〜 3.5 |
| 3 | dv1r | バーン1 ラジアルΔv (km/s) | 0 | -0.3 〜 0.3 |
| 4 | dv2r | バーン2 ラジアルΔv (km/s) | 0 | -0.3 〜 0.3 |
| 5 | azimuth | 打ち上げ方位角 (rad) | 0 | -0.2 〜 0.2 |
| 6 | b1h | バーン1 有効時刻 (hours) | 0.82 | 0.6 〜 1.1 |
| 7 | b2h | バーン2 有効時刻 (hours) | 1.8 | 1.5 〜 2.2 |
| 8 | peri_scale | 投入軌道近地点スケール | 1.0 | 0.85 〜 1.15 (±15%) |
| 9 | apo_scale | 投入軌道遠地点スケール | 1.0 | 0.85 〜 1.15 (±15%) |

投入軌道は公開値27×2222kmをベースに±15%の微調整を許容。
以前は投入軌道パラメータを完全に自由にしていたため非物理的な値（近地点-200km）に収束していた。
±15%制約により投入軌道が公開値の範囲に留まる。

### 制約（合わせるべき値）

ハード制約（6個）: Horizonsデータの最初の点の状態ベクトル（x, y, z, vx, vy, vz）
ソフト制約（4個）: 投入軌道 ≈ 27×2222km、バーン時刻 ≈ T+49min/T+108min

### コスト関数

2段階最適化:
- Stage 1: 位置合わせ重視 `cost = pos_err + vel_err * 1000`
- Stage 2: 速度方向重視 `cost = pos_err + vel_err * 2000`

Stage 1の結果をStage 2の初期値とする。

### 物理モデル

- 地球点質量重力
- J2（地球扁平率、1.08263×10⁻³）— 3.4hで~400kmの効果
- 月の重力（3体問題、潮汐力項含む）
- RK4積分、10秒ステップ（1秒も検証済み、改善なし）。Numba @njitでJITコンパイル済み（32倍高速化、77秒→2.4秒）
- インパルスバーン（瞬間的な速度変化、バーン時刻は有効中心として最適化）
- Phase 1: burn1からのRK4逆伝播（J2込み、Phase 2+3と同じ物理モデル）

含まれていないもの:
- J3（地球重力場3次項）: 3.4hで~4km。残差44kmの9%
- J4以降: 3.4hで~1km以下
- 太陽重力: 3.4hで~0.04km。無視できる
- 大気抵抗: 近地点通過時のみ。速度ロス~0 m/s
- 有限バーン時間: 遠地点上昇は18分間燃焼だがインパルス近似。バーン時刻最適化で有効中心として吸収
- 太陽輻射圧、相対論補正、地球の潮汐変形: すべて~0.01km以下

### 残差44kmの原因調査

| 候補 | 推定影響 | 検証 |
|------|---------|------|
| J3+J4 高次重力場 | ~5 km | 計算で確認、残差の11% |
| dt分解能(10s→1s) | 0 km | テスト済み、改善なし |
| 太陽重力 | ~0.04 km | 無視できる |
| 有限バーン時間 | 不明 | 未検証。18分間のバーン中の軌道変化 |
| Horizonsのモデル定数との差 | 不明 | GM値の桁数、地球半径の定義など |
| 投入軌道の実パラメータのずれ | 不明 | SpaceNewsで27×2222km確認済み。±30%テストでvel=0.1°だが近地点-123km |
| 軌道面のJ2歳差 | ~43km (近地点引数) | 3.4hで近地点引数が0.29°回転。burn1での43kmの位置ずれに相当 |

結論: 44kmは複数の小さな要因の積み重ね。単一の原因ではない。

### 解決済みの問題の経緯

#### 接続ジャンプの解決過程
1. 逆伝播（backward RK4）→ 地球の反対側にジャンプ（10,000km+）
2. Lambert解法 → 2体問題の限界で7,200kmジャンプ
3. 2バーン射撃法（グリッド探索）→ 913km
4. scipy最適化（5パラメータ）→ 47km（速度方向3.7°ずれ）
5. 9パラメータ最適化（J2なし）→ 0km/0°
6. propagate()と出力コードの2区間構造を一致 → 0km確認
7. 投入軌道固定+J2有効化+2段階最適化 → **44km/0.7°（現在）**

#### 投入軌道パラメータの改善過程
1. 完全自由（制約なし）→ 近地点-200km（非物理的）、接続0km
2. 近地点制約追加 → 近地点27km、遠地点3778km、接続0km
3. 近地点+遠地点制約 → 27km/2222km一致、速度方向3.2°
4. 固定27×2222km → 接続0km、速度4.6°
5. ±15%微調整+J2+2段階最適化 → **31km/2555km、接続44km/0.7°（現在）**

#### J2有効化の経緯
1. 最初の実装 → 有効化すると41km悪化（J2なし0km→J2あり41km）
2. 原因: 投入軌道パラメータが自由すぎてJ2の効果を吸収できなかった
3. 投入軌道固定+2段階最適化の導入後に再テスト → 安定動作
4. J2なしvs J2あり: 速度方向1.8°→0.7°に改善（位置は0km→44km）

#### 逆伝播アプローチの試行と断念
Phase 3（burn2→Horizons）を逆伝播で物理的に正確にするアプローチを試行。
- Phase 3は正確になったが、Phase 0-2の打ち上げ方向が-24〜-38°暴走
- 順方向+逆方向のハイブリッドも試行したが改善せず
- 原因: 逆伝播で得たburn2状態と、順方向の投入軌道+バーンの整合性が取れない
- 結論: 全区間順方向が最も安定

### ファイル構成

- `gen_trajectory.py`: 軌道合成スクリプト（synthesize_early_trajectory関数）
  - scipy.optimize.minimize (Nelder-Mead) による2段階最適化
  - RK4積分（地球+月+J2重力、10秒ステップ）
  - ケプラー方程式ソルバー（投入軌道Phase 1用）
  - .venv内のscipy+numba必要（`uv venv && uv pip install scipy numba`）
- `trajectory.json`: 生成された軌道データ（合成+Horizons結合）
- `data/horizons/`: JPL Horizonsの生データ
  - `fetch.sh`: 再取得スクリプト
  - `spacecraft.txt`, `moon.txt`: 10分間隔データ（全ミッション）
  - `spacecraft_fine.txt`, `moon_fine.txt`: 1分間隔データ（最初の28h）
- `debug_burns.py`: バーンデバッグ用スクリプト
